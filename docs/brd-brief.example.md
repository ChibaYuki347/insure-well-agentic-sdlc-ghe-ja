# ビジネスブリーフ（サンプル）— BRD作成エージェント用の入力例

> **用途:** `0.SDLC BRD Author` エージェントに渡す「ビジネスアイデア」の例です。
> これを元にエージェントがインタビュー（不足情報の確認）→ `docs/BRD.md` を作成します。
> **任意ステップ:** [handbook/guides/9.BRD-From-Scratch.ja.md](../handbook/guides/9.BRD-From-Scratch.ja.md)
>
> ⚠️ これは**入力例**です。そのまま使っても、自分のアイデアに差し替えてもOK。完成版の
> BRD ではありません（完成版の参考は [docs/BRD.md](BRD.md)）。

---

## 例1: 契約者セルフサービス・ポータル（推奨）

> InsureWell の契約者が、保険の確認・請求・通知をオンラインで完結できる
> セルフサービス体験を提供したい。現状はデモ用の最小機能（ポリシー/請求の閲覧・編集）
> しかなく、認証も通知も無いため、実利用に耐えない。
>
> - **狙い:** サポート問い合わせを減らし、請求の透明性を高める。
> - **対象:** 契約者（自分のポリシー/請求のみ）、管理者（請求の審査・対応）。
> - **当面の範囲:** ログイン/権限、請求履歴とステータスの可視化、ステータス変更時の通知。
> - **対象外（初期）:** 決済、SSO、モバイルアプリ。
> - **制約:** React + Spring Boot、デモはローカル（H2）。将来の本番移行は見据える。
> - **成功指標(案):** 主要操作の認証必須化100%、請求状況の自己確認が可能、通知の到達。

エージェントへの渡し方（例）:

```
@0.SDLC BRD Author 次のビジネスアイデアから BRD を作りたいです。
不足があればインタビューしてください。
[上の「例1」を貼り付け]
```

## 例2: 管理者向け請求オペレーション・ダッシュボード

> 管理者/査定担当が、請求キューを効率よくさばくための運用ダッシュボードを作りたい。
> 検索・絞り込み・メモ・状態遷移の記録（監査）を備え、対応漏れを防ぐ。
>
> - **対象:** 査定担当、オペレーションマネージャー。
> - **範囲:** 請求一覧の検索/絞り込み、請求詳細でのメモと状態変更、変更履歴の記録。
> - **対象外（初期）:** 高度な分析、外部連携。
> - **成功指標(案):** 平均処理時間の短縮、対応漏れ件数の可視化。

## 例3: 自由記述（自分のアイデアで）

> 上の例にとらわれず、自分の業務に近いテーマで1〜2段落のアイデアを書いてエージェントに
> 渡してください。情報が足りなければエージェントが質問します。

## エージェントに聞かれやすいこと（事前に考えておくと速い）

- 解決したい課題は何か／なぜ今か
- 対象ユーザー・ロール（社内/社外）
- 測定可能なビジネスゴール（成功の定義）
- スコープ（初期リリースの In / Out）
- 制約・前提（規制、データ、技術スタック、デモ/本番）
- 優先度の付け方（P1/P2/P3 の基準）
