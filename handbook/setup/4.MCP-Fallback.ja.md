# MCP フォールバック（MCPなしでも進める）

> **このワークショップで MCP は「加速装置」であって必須ではありません。** MCP を使うのは主に
> 2か所だけで、いずれも**非MCPの代替**があります。設定が間に合わない・社内制約で使えない・
> リスクを避けたい場合は、このページの代替経路でそのまま進められます。
> **ローカルMCP設定:** [3.MCP-Local-Setup.ja.md](3.MCP-Local-Setup.ja.md) ・ **事前作成Issue:** [../docs/seed-issues.md](../../docs/seed-issues.md)

---

## 1. MCP はどこで使う? どこは不要?

| 工程 | MCP を使う場合 | **非MCPの代替** |
| --- | --- | --- |
| Issue 作成 | GitHub MCP（`@github` で作成）| **Web UI / `gh` CLI / 事前作成スクリプト** |
| クラウドエージェントへ委任 | （任意）| **github.com で Issue を Copilot に割り当て＝そもそもMCP不要** |
| コーディング（ローカル）| 不要 | VS Code の Copilot Agent Mode（MCP不要）|
| コードレビュー | 不要 | PR 画面の **Copilot review**（MCP不要）|
| GHAS | 不要 | Security タブ（MCP不要）|
| UI 検証（Playwright）| Playwright MCP（対話的にブラウザ操作）| **`npm run test:e2e`（@playwright/test を直接実行）** |

> ポイント: **コーディング・レビュー・GHAS は元々 MCP 非依存**。MCP が無くて困るのは
> 「Issue 作成」と「Playwright の対話操作」だけで、どちらも代替があります。

## 2. Issue 作成（MCPなし）

### 2-A. 一括スクリプト（推奨・速い）

`gh` CLI で事前作成Issueをまとめて作成します（文面は [../docs/seed-issues.md](../../docs/seed-issues.md)）。

```bash
# まず gh にログイン
gh auth login

# 中身を確認（API を呼ばない）
./scripts/create-seed-issues.sh --repo <owner>/<name> --dry-run

# 実行（5件作成）
./scripts/create-seed-issues.sh --repo <owner>/<name>
```

> リポジトリ名は引数 `--repo` か環境変数 `REPO` で指定します（個社リポジトリ名はハードコードしていません）。

### 2-B. 単発の `gh` コマンド

```bash
gh issue create --repo <owner>/<name> \
  --title "User authentication & role-based authorization" \
  --label "enhancement,backend,frontend,auth,P1" \
  --body-file - <<'EOF'
**As a** policyholder, **I want to** log in ... (本文)
EOF
```

### 2-C. 完全手動（Web UI）

github.com → 対象リポジトリ → **Issues** → **New issue** に、
[../docs/seed-issues.md](../../docs/seed-issues.md) の Title / Body / Labels を貼り付け。

## 3. クラウドエージェントへの委任（MCPなし）

委任は **github.com 側の操作**で完結し、ローカル MCP は不要です。

1. 対象 Issue を開く。
2. 右側の **Assignees** で **Copilot** を割り当てる（または Issue から「Let Copilot work on this」）。
3. Copilot がブランチを作り、実装して **PR を起票**するのを待つ。

> これが「非同期で走らせる」本命。昼休み前に着手しておけば、MCP の有無に関係なく進みます。

---

## 4. UI 検証（Playwright・MCPなし）

Playwright MCP の対話操作の代わりに、**テストスイートを直接実行**します。
既存のテストは `src/frontend/tests/` にあります。

```bash
cd src/frontend
npm install                 # 初回のみ
npx playwright install chromium   # ブラウザ未導入なら

npm run test:e2e            # 全テスト
npm run test:e2e:headed     # ブラウザ表示で実行
npm run test:e2e:dashboard  # ダッシュボードのみ
npm run test:e2e:claims     # 請求のみ
npm run test:e2e:report     # HTML レポートを表示
```

> QA エージェント（`@6.SDLC Test Agent`）でのテスト**生成**自体は MCP 非依存です。
> 生成したテストをこのコマンド群で実行すれば、Playwright MCP が無くても UI 検証は成立します。

## 5. 当日の判断フロー（講師向け）

```
MCP のツールが Chat に出ている?
├─ はい → 通常フロー（@github で Issue 作成 / Playwright MCP で対話検証）
└─ いいえ →
     ├─ Issue 作成     → scripts/create-seed-issues.sh（または Web UI / gh）
     ├─ 委任           → github.com で Copilot に割り当て（元々 MCP 不要）
     ├─ コーディング/レビュー/GHAS → そのまま（MCP 不要）
     └─ UI 検証        → npm run test:e2e（直接実行）
```

> **止めない原則:** MCP に時間を溶かさない。出なければ代替へ切り替えて先へ進む。
> MCP の再設定は休憩時間や宿題に回せます（環境は後日も利用可）。

## 6. MCP を使うときの留意（リスク低減）

- **PAT は最小スコープ**（`repo` / `read:org` / `workflow`）。期限は短め（数日）。終了後に失効させる。
- **トークンはファイルに書かない** — `.vscode/mcp.json` は `${input:github-pat}` で起動時入力。
- **ローカル限定**（リモート MCP は使わない）。取得は社内プライベートレジストリ（`.npmrc`）。
- エージェントがツールで行う書き込み操作（Issue/PR 作成等）は**内容を確認してから**承認する。
