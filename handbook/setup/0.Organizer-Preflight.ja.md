# 主催者／管理者 事前準備チェックリスト（最重要・前日まで）

> **これは何?** 当日いちばん時間が溶けるのは「環境・権限まわり」です。特に **Cloud Agent
> （Copilot コーディングエージェント）・Copilot コードレビュー・GHAS は Organization 配下の
> リポジトリでないと動かせず**、Org 配下のリポジトリで Issue を作るには **Org スコープの PAT** が必要で、
> 組織ポリシーによっては **グローバル管理者の承認**が要ります。承認には**リードタイム**があるため、
> これらは必ず**前日まで**に主催者／管理者が段取りしてください。
> **参加者向けの前提:** [1.Prerequisites.ja.md](1.Prerequisites.ja.md) ・ **進行台本:** [../Instructor-Runbook.ja.md](../Instructor-Runbook.ja.md)

---

## 0. なぜ Organization 配下のリポジトリが必要か

| 機能 | 個人リポジトリ | **Organization 配下** |
| --- | --- | --- |
| Copilot **コーディングエージェント**（Issue を Copilot に委任）| 制限あり/不可の場合あり | ✅ ポリシーで有効化して利用 |
| Copilot **コードレビュー**（PR 自動レビュー）| 制限あり | ✅ 組織で有効化 |
| **GHAS**（Code scanning / Secret scanning / Dependabot）| Public は一部のみ | ✅ ライセンス＋有効化で全機能 |

> 当日のゴール（Issue→委任→PR→レビュー→GHAS）を成立させるには、**Org 配下リポジトリ＋下記の
> 有効化・権限**が前提になります。

---

## A. リポジトリと Copilot

- [ ] ワークショップ用リポジトリを **Organization 配下**に用意（参加者がアクセスできる）。
- [ ] 参加者全員に **Copilot のシート（ライセンス）** が割り当て済み。
- [ ] **Copilot コーディングエージェント**を Org/リポジトリのポリシーで**有効化**
      （Issue を Copilot に割り当てて PR を作らせる機能）。
- [ ] **Copilot コードレビュー**を有効化（PR で自動レビューが走る設定 / ルールセット）。

## B. GHAS（GitHub Advanced Security）

- [ ] 対象リポジトリで **GHAS を有効化**（プライベートの場合はライセンスが必要）。
- [ ] **Code scanning（CodeQL）** … *Default setup* を有効化（言語: Java / JavaScript）。
- [ ] **Secret scanning**（＋ push protection）を有効化。
- [ ] **Dependabot**（alerts / security updates）を有効化。

## C. 権限・PAT（いちばん詰まるポイント）

- [ ] 参加者が Issue を作る方法を**事前に確定**（下記のいずれか）。
  - **方式1: 参加者ごとに Org スコープの PAT**
    - Fine-grained PAT、Resource owner = **対象 Org**、Repository permissions に
      **Issues: Read and write**（必要に応じ Contents / Pull requests も）。
    - ⚠️ Org が「**Fine-grained PAT の承認制**」を有効にしている場合、**管理者が各トークンを承認**するまで
      使えません（**リードタイムが発生**）。前日までに承認フローを回しておく。
  - **方式2: 事前に Issue を中央作成（推奨の保険）**
    - ファシリテーター（書き込み権限あり）が **`scripts/create-seed-issues.sh`** で
      バックログ Issue を**事前に作成**しておく → 参加者は PAT 承認を待たずに着手可能。
    - 参照: [../../docs/seed-issues.md](../../docs/seed-issues.md) / [4.MCP-Fallback.ja.md](4.MCP-Fallback.ja.md)
- [ ] **承認の責任者（グローバル/Org 管理者）と所要時間**を把握（誰に・いつ依頼するか）。

## D. ブランチ保護 / 必須チェック（レビュー〜マージのデモ用）

- [ ] `main` に**ブランチ保護 / ルールセット**を設定（PR 必須・必要なら必須レビュー）。
- [ ] CI / Code scanning を**必須ステータスチェック**に（マージゲートのデモが映える）。
      ※ ただし当日詰まらないよう、**厳しすぎる必須化は避ける**（バイパス手順も用意）。

## E. ネットワーク / MCP（ローカル）

- [ ] 会場ネットワーク/プロキシで **github.com** と **社内プライベートレジストリ**へ到達可能。
- [ ] ローカル MCP（`.vscode/mcp.json` / `npx`）は同梱済み。設定は [3.MCP-Local-Setup.ja.md](3.MCP-Local-Setup.ja.md)。

## 承認リードタイム早見表（前日までに段取り）

| 項目 | 承認/設定する人 | 目安リードタイム | 未完だと当日… |
| --- | --- | --- | --- |
| Org 配下リポジトリの用意・アクセス付与 | Org 管理者 | 即〜1日 | そもそも委任/レビュー/GHAS 不可 |
| Copilot シート割当 | Copilot 管理者 | 即〜1日 | エージェントが使えない |
| コーディングエージェント有効化 | Org 管理者 | 即〜1日 | Issue を委任できない |
| GHAS 有効化（ライセンス） | Org/Security 管理者 | 1日〜 | GHAS フェーズができない |
| **Fine-grained PAT の承認** | **グローバル/Org 管理者** | **数時間〜数日** | **Issue 作成で全員ブロック** |

> ⏱ **最大のリスクは「PAT 承認待ち」。** 承認制の Org では、当日朝に全員分が一気に承認待ちになり
> 進行が止まります。**方式2（Issue 中央事前作成）を保険**として必ず用意してください。

## 検証コマンド（前日に主催者が実機確認）

```bash
# gh CLI の認証状態
gh auth status

# 対象リポジトリが Org 配下で見えるか
gh repo view <org>/<repo> --json nameWithOwner,visibility,isInOrganization

# Issue を作成できる権限があるか（dry-run はスクリプト側で）
./scripts/create-seed-issues.sh --repo <org>/<repo> --dry-run

# GHAS / code scanning の有効状態（要権限）
gh api repos/<org>/<repo>/code-scanning/alerts --jq 'length' 2>/dev/null \
  || echo "code scanning 未有効 or 権限不足"

# Copilot コーディングエージェントは UI（Issue の Assignees に Copilot が出るか）で確認
```

## 当日朝の最終チェック（主催者）

- [ ] 参加者がリポジトリを clone でき、`./run.sh` で起動する
- [ ] 参加者が **Issue を作成できる**（PAT 承認済み or 中央事前作成済み）
- [ ] Issue の Assignees に **Copilot** が出る（委任できる）
- [ ] PR を作ると **Copilot レビュー**と **Code scanning** が走る
- [ ] 事前作成 Issue / 参照ブランチ（[../Instructor-Runbook.ja.md](../Instructor-Runbook.ja.md) §3）が用意済み
