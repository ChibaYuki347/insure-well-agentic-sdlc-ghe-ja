# 概念プライマー — はじめに押さえる用語

> **対象:** 本編に入る前の 30〜35 分で講師が解説する基礎概念のまとめ。
> ハンズオンの前に「何を・なぜ・どの仕組みで」やるのかを共有します。
> **English agenda:** [Agenda.md](../Agenda.md) ・ **進行台本:** [Instructor-Runbook.ja.md](../Instructor-Runbook.ja.md)

このワークショップは、**GitHub Copilot のエージェント**を使って、要件定義からコーディング・
テスト・レビューまで、ソフトウェア開発の各フェーズを自動化する流れを体験するものです。
題材は React + Spring Boot の保険管理アプリ **InsureWell** です。

---

## 1. Agentic SDLC（エージェント駆動の開発ライフサイクル）とは

従来の SDLC（要件 → 設計 → 実装 → テスト → レビュー → リリース）の各工程を、
**人がエージェントに委任（delegate）** して進める進め方です。

```
Issue 作成  →  クラウド/コーディングエージェントへ委任  →  PR 自動生成
   →  Copilot 自動コードレビュー  →  CI/CD  →  GHAS（セキュリティ）  →  QA  →  マージ
```

- 人は「**何を作るか（What）**」の入力（要件・受け入れ条件）に集中します。
- エージェントが「**どう作るか（How）**」のコードを書きます。**参加者は一行もコードを書きません。**
- 大事なのは「全部を完成させること」ではなく、**新しい働き方の型を体感すること**です。
  途中で詰まっても、環境は後日も使えるので各自で続きを試せます。

> 図解は [3.Understand_Workflow.md](3.Understand_Workflow.md) を参照。

## 2. BRD（Business Requirements Document）とは

**BRD = ビジネス要件定義書。** 「何を・なぜ作るのか」をビジネス視点で言語化した文書です。
コードやアーキテクチャ（How）ではなく、**目的・課題・ゴール・対象ユーザー・スコープ・優先度**を書きます。

| 含まれるもの | 例（InsureWell） |
| --- | --- |
| 目的 / 課題 | 認証が無く誰でも保険・請求を編集できる |
| ビジネスゴール | 役割に応じた安全なアクセス（G1）など |
| ペルソナ | 契約者 / 査定担当 / 管理者 / サポート |
| スコープ（In/Out） | 認証は In、決済は Out |
| 優先度付きバックログ | P1: 認証, P1: 請求履歴, P2: レポート… |

本リポジトリの実例は [docs/BRD.md](../../docs/BRD.md)。

**ワークショップでの位置づけ:** BRD エージェント（`@1.SDLC BRD Agent`）が既存アプリを分析し、
ギャップを洗い出して**優先度付きバックログ**を生成 → そこから GitHub Issue を作成します。
**Issue が良ければ実装は安定します。** だからこそ要件（入力）にいちばん時間をかけます。

## 3. MCP（Model Context Protocol）とは

### 3.1 原理 — 何が変わるのか

MCP は、Copilot エージェントに **外部サービスへの構造化されたアクセス手段（ツール）** を与える
オープンな規格です。

- **MCP なし:** Copilot はファイルの読み書きしかできません。
- **MCP あり:** Issue 作成・PR 起票・ブランチ操作・コード検索・ブラウザ操作などを
  「ツール呼び出し」として実行できます。

```
Copilot エージェント  ──(ツール呼び出し)──►  MCP サーバー  ──►  GitHub / ブラウザ など
```

代表例:
- **GitHub MCP サーバー** — Issue / PR / ブランチ / コード検索
- **Playwright MCP サーバー** — ブラウザ自動操作（UI 検証）

### 3.2 種類 — ローカル MCP と リモート MCP

| 種類 | 動く場所 | 例 |
| --- | --- | --- |
| **ローカル MCP** | 自分の PC 上のプロセス（`stdio` / `npx` 起動） | `.vscode/mcp.json` から `npx` で起動 |
| **リモート MCP** | ネットワーク越しの外部サーバー（`http`/SSE） | クラウドにホストされた共有 MCP |

> ⚠️ **重要な制約:** 社内ポリシーにより**リモート MCP の利用は禁止**です。
> 本ワークショップは **すべてローカル MCP** で行います。MCP は `.vscode/mcp.json` に定義し、
> **社内プライベートレジストリ**から `npx` でパッケージを取得してローカル起動します。
> 具体的な設定は [setup/3.MCP-Local-Setup.ja.md](../setup/3.MCP-Local-Setup.ja.md) を参照。

### 3.3 スコープ（ディスカバリ）— どこに登録するか

同じ MCP サーバーでも、**どこに設定ファイルを置くか**で適用範囲（discovery scope）が変わります。

| スコープ | 設定場所 | 適用範囲 | 使いどころ |
| --- | --- | --- | --- |
| **ワークスペース** | `<repo>/.vscode/mcp.json` | そのリポジトリのみ | チームで共有（本ワークショップの基本） |
| **ユーザー（VS Code）** | VS Code のユーザー設定 | その VS Code 全体 | 個人の常用ツール |
| **マシン** | OS/ユーザー全体の設定 | その PC 全体 | 全プロジェクト共通 |

> 本ワークショップでは **ワークスペーススコープ**（リポジトリ直下の `.vscode/mcp.json`）を使い、
> 全員が同じ構成・同じソースで揃うようにします。

## 4. カスタムエージェントとは

`.github/agents/` 配下に置く `.agent.md` ファイルで定義する、**役割特化のエージェント**です。
チームの専門家のように、focused な役割と専用の指示を持ちます。Copilot Chat から名前で呼び出します。

本リポジトリに用意済みのエージェント:

| エージェント | 役割 |
| --- | --- |
| `1.SDLC BRD Agent` | ビジネス要件（BRD・エピック・フィーチャー）|
| `2.SDLC HLD Agent` | 高位設計（アーキテクチャ・コンポーネント境界・データフロー）|
| `5.SDLC Dev Agent` | 実装（React + Spring Boot）|
| `6.SDLC Test Agent` | テスト生成・CI トリアージ・カバレッジ |

終盤の演習では、**自分でカスタムエージェントを作る**ところまで体験します
（[7.Custom-Agents-Exercise.ja.md](7.Custom-Agents-Exercise.ja.md)）。

## 5. 委任モード（どこから仕事を始めるか）

| モード | いつ使う | 仕組み |
| --- | --- | --- |
| **ローカルエージェント**（VS Code Agent Mode）| 対話的なコーディング・デバッグ | ワークスペース/ターミナル/未コミット変更を見て、各ステップを操作 |
| **クラウドエージェント**（VS Code から委任）| 作業中に見つけたタスクを非同期で渡す | 裏で実行し、結果を PR として返す |
| **GitHub Issue → Copilot** | バックログ化された作業 | github.com で Issue を Copilot に割り当て → Issue から実装 |

合言葉: **ローカル = 対話・今すぐ / クラウド = 非同期で委任 / Issue = チームで追跡。**
詳細は [4.Copilot-Agent-Delegation-Guide.md](4.Copilot-Agent-Delegation-Guide.md)。

## 6. このあとの流れ

1. 環境スモークチェック（前提は事前課題で完了済み）
2. ローカル MCP 設定 + カスタムエージェント概要
3. 要件 & 設計（HLD → BRD → バックログ）
4. Issue 作成（**当日扱う機能を 1 つに絞る**）
5. クラウドエージェントへ委任（昼前に着手 → 昼休み中に非同期実行）
6. コーディング → コードレビュー → GHAS → QA + Playwright
7. カスタムエージェント演習 → まとめ

全体スケジュールは [Agenda.ja.md](../Agenda.ja.md) を参照してください。
