# InsureWell — Agentic SDLC デモフロー

> このウォークスルーでは、VS Code 内の **GitHub Copilot エージェント**を使った完全な Agentic SDLC を体験します。
> InsureWell リポジトリ（React + Spring Boot）をベースに、要件定義から PR マージまでを一気通貫で進めます。

---

## ステップ 1 — アプリをクローン・ビルド・起動する

```bash
# 1. リポジトリをクローン
git clone https://github.com/im-naga-ghas/insure-well-agentic-sdlc-ghe.git
cd insure-well-agentic-sdlc-ghe

# 2. バックエンドとフロントエンドを起動
cd src
chmod +x run.sh
./run.sh
```

ブラウザで **http://localhost:3000** を開いてください。

初回起動時、Spring Boot バックエンドは **http://localhost:8080/api** でサンプルデータ（ポリシー・請求）が入った **H2** インメモリDBで起動します。React フロントエンドはポート **3000** で動作します。

**画面で確認できること：**
- **Dashboard** — ポリシー管理、ライフサイクルステータス、ポリシー別統計、最近の請求
- **Claims** — 請求フォーム、ポリシー別フィルター、ステータス追跡（保留中 / 承認 / 却下）

## ステップ 2 — GitHub MCP サーバーを有効にする

**GitHub MCP（Model Context Protocol）サーバー**により、Copilot エージェントが GitHub API（Issue、PR、ブランチ、コード検索）に直接アクセスできるようになります。

> ⚠️ **このセッションはローカル限定**（リモート MCP は禁止）。
> ローカルの `.vscode/mcp.json` テンプレートを使用してください。
> → [setup/3.MCP-Local-Setup.ja.md](../setup/3.MCP-Local-Setup.ja.md)

1. VS Code コマンドパレットを開く（`Cmd+Shift+P` / `Ctrl+Shift+P`）。
2. **「MCP: Add Server」** を実行し、**GitHub** を選択する。
3. プロンプトが表示されたら認証する（GitHub トークンまたは OAuth フロー）。
4. ステータスバーの MCP インジケーターが緑色になれば起動完了。

> 代替手段 — `.vscode/mcp.json` に手動追加：
> ```json
> {
>   "servers": {
>     "github": {
>       "type": "stdio",
>       "command": "npx",
>       "args": ["-y", "@modelcontextprotocol/server-github"],
>       "env": {
>         "GITHUB_PERSONAL_ACCESS_TOKEN": "<YOUR_PAT>"
>       }
>     }
>   }
> }
> ```
> 必要なトークンスコープ：`repo`、`read:org`、`read:user`

## ステップ 3 — Agentic SDLC 用のカスタムエージェントを追加する

カスタムエージェントは `.github/agents/`（または `.vscode/agents/`）内の `.agent.md` ファイルとして定義します。各エージェントは専用の役割と指示・ツールアクセスを持ちます。

**InsureWell 推奨エージェントセット：**

| エージェントファイル | 役割 |
|---|---|
| `2.SDLC HLD Agent` | 高レベル設計 — アーキテクチャ、コンポーネント境界、データフロー |
| `1.SDLC BRD Agent` | ビジネス要件 — BRD、エピック、フィーチャー |
| `5.SDLC Dev Agent` | フィーチャー実装 — React + Spring Boot |
| `cloud-agent.md` | クラウド/インフラ — IaC、デプロイ設定 |
| `6.SDLC Test Agent` | QA — テスト生成、CIトリアージ、カバレッジ |
| `review-agent.md` | コードレビュー — PR レビュー、標準、セキュリティ確認 |

エージェントを作成するには、Copilot Chat パネルを開き **Agent モード**に切り替えてから：

```
/create-agent
```

または `.github/agents/` 配下に `applyTo` パターンとシステム指示を含む `.agent.md` ファイルを直接作成します。

## ステップ 4 — HLD エージェントで高レベル設計を生成する

> 🆕 **オプション — ステップ 0: ビジネスアイデアから BRD を作成する。** このウォークスルーでは
> [docs/BRD.md](../../docs/BRD.md) が存在することを前提としています。ない場合、または要件定義の
> 練習をしたい場合は、まずインタビュー形式の **`@0.SDLC BRD Author`** エージェントで
> `docs/BRD.md` を生成してから続けてください。詳細は [9.BRD-From-Scratch.ja.md](9.BRD-From-Scratch.ja.md) と
> サンプルブリーフ [docs/brd-brief.example.md](../../docs/brd-brief.example.md) を参照。

Copilot Chat を **Agent** モードで開き、HLD エージェントを呼び出します：

```
@2.SDLC HLD Agent InsureWellアプリケーションの高レベル設計を生成してください。
含めるもの: コンポーネント図、ReactフロントエンドとSpring Boot REST APIの間のデータフロー、
H2/JPAパーシステンスレイヤー、請求の提出/更新APIサーフェス。
```

**期待する出力：**
- アーキテクチャ概要（React フロントエンド、Spring Boot バックエンド、JPA リポジトリ、H2 データベース）
- コンポーネント相互作用図（Mermaid または ASCII）
- データフロー：ブラウザ → React コンポーネント → REST API → サービス/コントローラー/リポジトリ → DB → JSON レスポンス
- `PolicyController.java` と `ClaimController.java` に沿った API サーフェスマップ

HLD を確認・修正した後、`docs/InsureWell_HLD.md` に保存または更新してください。

## ステップ 5 — BRD エージェントで要件を生成する

BRD エージェントを使って既存アプリを分析し、新しいユーザーストーリーを生成します：

```
@1.SDLC BRD Agent InsureWellアプリケーション（src/backend、src/frontend、README.md）を分析してください。
現在の機能セットのギャップを特定し、アプリを拡張するための
ビジネス要件、エピック、ユーザー向けフィーチャーの優先順位付きバックログを作成してください。
```

**期待する出力：**
- 既存機能のまとめ（Phase 1 MVP）
- 特定されたギャップ（例：認証、通知、レポート、分析）
- BRD ゴール、エピック、フィーチャーにマッピングされた優先順位付きバックログ
- HLD エージェントと Dev エージェントに渡せる受け入れ指向のフィーチャー文

## ステップ 6 — 新しいユーザーストーリーを作成する

BRD エージェントの出力から、上位ストーリーの GitHub Issue を作成します。バックログのシードとなる例：

| # | ストーリー | 優先度 |
|---|---|---|
| 1 | 契約者として、ユーザー名とパスワードでログインし、ポリシーをプライベートにしたい | 高 |
| 2 | 契約者として、請求ステータスが変わったときにメール通知を受け取り、常に把握したい | 高 |
| 3 | 管理者として、請求トレンドと承認率のレポートダッシュボードで運用を監視したい | 中 |
| 4 | 契約者として、ポリシーのPDFサマリーをダウンロードし、オフラインで記録したい | 中 |
| 5 | 管理者として、ステータスやプランタイプでポリシーを検索・フィルターし、効率的に管理したい | 中 |
| 6 | 契約者として、請求ステータス変更のタイムラインを確認し、審査プロセスを理解したい | 低 |

Copilot Chat から GitHub MCP サーバー経由で Issue を作成します：

```
@github 以下のユーザーストーリーのGitHub Issueを
im-naga-ghas/insure-well-agentic-sdlc-gheに適切なラベル
（enhancement、frontend、backend、auth）付きで作成してください：
[ストーリーリストを貼り付ける]
```

## ステップ 7 — ストーリーをクラウドエージェントまたは Dev エージェントに割り当てる

バックログからストーリーを 1 つ選びます（例：*ストーリー 1 — ユーザー認証*）。

**オプション A — Dev エージェント（フィーチャー実装）：**

```
@5.SDLC Dev Agent InsureWellにユーザー認証を実装してください（ストーリー #1）。
- Spring Boot バックエンドに Spring Security ベースのログインサポートを追加
- JPA/H2 に適したユーザーモデルと永続化戦略を導入
- React フロントエンドにサインイン/サインアウトの認証 UI フローを追加
- 認証/認可の背後にクレームとポリシー管理アクションを保護
- デモ用にデフォルトのローカル管理者ユーザーをシード
- 認証ワークフローのバックエンドとフロントエンドのテストを追加
```

**オプション B — クラウドエージェント（インフラ/デプロイ）：**

```
@cloud-agent InsureWellのデプロイ設定を作成してください。
- Spring Boot バックエンドと React フロントエンドの Dockerfile
- ローカル/デモ環境でWebアプリとAPIを一緒に動かすdocker-compose.yml
- main へのプッシュ時にコンテナイメージをビルド・公開するGitHub Actionsワークフロー
- Spring プロファイル、API ベース URL、デプロイ設定の環境変数とシークレット管理
```

エージェントはフィーチャーブランチを作成し、変更を実装してプルリクエストを開きます。

## ステップ 8 — QA エージェントでテストを追加し、CI の失敗を調査してカバレッジを確認する

Dev またはクラウドエージェントが PR をオープンした後、QA エージェントを呼び出します：

**テストを追加する：**
```
@6.SDLC Test Agent ストーリー #1（ユーザー認証）のPRをレビューしてください。
以下をカバーするテストを生成してください：
- ログイン成功と認証済みAPIアクセス
- 間違ったパスワードでのログイン失敗
- 認証なしでの保護されたルートまたはアクションへのアクセス
- ログアウト/セッション無効化の動作
バックエンドテストは `src/backend/src/test/`、フロントエンドテストは `src/frontend/src/` に追加してください。
```

**CI の失敗を調査する：**
```
@6.SDLC Test Agent PR #<番号> のCIパイプラインが失敗しています。
GitHub Actions のログを分析し、根本原因を特定して、
修正案を提案または適用してください。
```

**コードカバレッジを確認する：**
```
@6.SDLC Test Agent カバレッジレポート付きでバックエンドとフロントエンドのテストスイートを実行し、
80%未満のモジュールやブランチを特定してください。
ギャップを埋めるための追加テストケースを提案してください。
```

---

## ステップ 9 — レビューエージェントで PR をレビューする

テストがパスしたら、オープンな PR でレビューエージェントを呼び出します：

```
@review-agent ユーザー認証フィーチャーの PR #<番号> をレビューしてください。
以下をチェックしてください：
- セキュリティ上の問題（パスワードハッシュ、セッション処理、CSRF/CORS、認可の漏れ）
- 既存の Spring Boot と React コードベースとのコードスタイルの一貫性
- エラー処理やエッジケースの欠落
- ドキュメントとインラインコメント
- OWASP Top 10 の懸念事項
PR にインラインコメントとサマリー判定（承認 / 修正要求）を投稿してください。
```

レビューエージェントは GitHub MCP サーバーを通じて、プルリクエストに直接構造化されたレビューコメントを投稿します。

---

## フルデモフローのまとめ

```
クローン & 起動  →  GitHub MCP 有効化  →  カスタムエージェント追加
     ↓
HLD エージェント（アーキテクチャ）  →  BRD エージェント（要件 & バックログ）
     ↓
GitHub Issue を作成（ストーリー）  →  コーディング/クラウドエージェントに割り当て
     ↓
QA エージェント（テスト、CI トリアージ、カバレッジ）  →  レビューエージェント（PR レビュー）
     ↓
PR をマージ  →  次のストーリーへ繰り返す
```
