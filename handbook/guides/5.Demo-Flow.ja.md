# InsureWell — エージェント駆動 SDLC デモフロー

このウォークスルーでは、InsureWell リポジトリの新規クローンを起点に、VS Code 内の **GitHub Copilot エージェント** を活用したエージェント駆動ソフトウェア開発ライフサイクル (SDLC) の全体像を、現在の **React + Spring Boot** 実装に沿ってデモします。

---

## Step 1 — クローン、ビルド、アプリの起動

```bash
# 1. リポジトリをクローン
git clone https://github.com/im-naga-ghas/insure-well-agentic-sdlc-ghe.git
cd insure-well-agentic-sdlc-ghe

# 2. バックエンドとフロントエンドを起動
cd src
chmod +x run.sh
./run.sh
```

ブラウザで **http://localhost:3000** を開きます。

初回起動時、Spring Boot バックエンドは **http://localhost:8080/api** で起動し、サンプルの保険契約とクレームが投入された **H2** インメモリデータベースを使用します。React フロントエンドはポート **3000** で動作します。

**画面で見られるもの:**
- **Dashboard** — 保険契約管理、ライフサイクルステータス、契約ごとの統計、最近のクレーム
- **Claims** — クレーム提出フォーム、契約ベースのフィルタ、ステータス管理 (Pending / Approved / Rejected)

## Step 2 — GitHub MCP サーバーを有効化

**GitHub MCP (Model Context Protocol) サーバー** により、Copilot エージェントは Issue、Pull Request、ブランチ、コード検索などの GitHub API に直接アクセスできるようになります。

> ⚠️ **本セッションはローカル限定** (リモート MCP は禁止)。
> [setup/3.MCP-Local-Setup.ja.md](../setup/3.MCP-Local-Setup.ja.md) に従い、ローカルの
> `.vscode/mcp.json` テンプレートとプライベートレジストリを使用してください。

1. VS Code のコマンドパレットを開く (`Cmd+Shift+P` / `Ctrl+Shift+P`)。
2. **「MCP: Add Server」** を実行し **GitHub** を選択。
3. プロンプトに従って認証 (GitHub トークンまたは OAuth フローを使用)。
4. サーバーが動作していることを確認: ステータスバーの MCP ステータスインジケータが緑になります。

> もしくは `.vscode/mcp.json` に手動でサーバーエントリを追記:
> ```json
> {
>   "servers": {
>     "github": {
>       "type": "stdio",
>       "command": "npx",
>       "args": ["-y", "@modelcontextprotocol/server-github"],
>       "env": {
>         "GITHUB_PERSONAL_ACCESS_TOKEN": "<YOUR_PAT>"
>       }
>     }
>   }
> }
> ```
> 必要なトークンスコープ: `repo`、`read:org`、`read:user`。

## Step 3 — エージェント駆動 SDLC 用のカスタムエージェントを追加

カスタムエージェントは `.github/agents/` (または `.vscode/agents/`) 配下の `.agent.md` ファイルとして定義します。各エージェントはフォーカスされたロールと、それに合わせた指示・ツールアクセスをカプセル化します。

**InsureWell 向け推奨エージェントセット:**

| エージェントファイル | 役割 |
|---|---|
| `2.SDLC HLD Agent` | 高位設計 — アーキテクチャ、コンポーネント境界、データフロー |
| `1.SDLC BRD Agent` | ビジネス要件 — BRD、エピック、フィーチャー |
| `5.SDLC Dev Agent` | フィーチャー実装 — React + Spring Boot のデリバリー |
| `cloud-agent.md` | クラウド/インフラ — IaC、デプロイ構成、環境セットアップ |
| `6.SDLC Test Agent` | 品質保証 — テスト生成、CI 失敗分析、カバレッジ |
| `review-agent.md` | コードレビュー — PR レビュー、規約、セキュリティチェック |

これらのエージェントを作るには、Copilot Chat パネルを開き **Agent モード** に切り替えて以下を実行:

```
/create-agent
```

もしくは `.github/agents/` 直下に適切な `applyTo` パターンとシステム指示を持つ `.agent.md` ファイルを直接作成します。

## Step 4 — HLD エージェントで HLD を作成

Copilot Chat を **Agent** モードで開き、HLD エージェントを呼び出します:

```
@2.SDLC HLD Agent InsureWell アプリケーションの高位設計 (HLD) を作成してください。
コンポーネント図、React フロントエンドと Spring Boot REST API 間のデータフロー、
H2/JPA 永続化レイヤー、クレーム提出/更新の API サーフェスを含めてください。
```

**期待される出力:**
- アーキテクチャ概要 (React フロントエンド、Spring Boot バックエンド、JPA リポジトリ、H2 データベース)
- コンポーネント間のインタラクション図 (Mermaid または ASCII)
- データフロー: ブラウザ → React コンポーネント → REST API → サービス/コントローラ/リポジトリ → DB → JSON レスポンス
- `PolicyController.java` と `ClaimController.java` に整合した API サーフェス

HLD をレビュー・洗練し、リポジトリの `docs/InsureWell_HLD.md` に保存または更新します。

## Step 5 — BRD エージェントで要件を作成

BRD エージェントで現状アプリを分析し、新しいユーザーストーリーを生成します:

```
@1.SDLC BRD Agent InsureWell アプリケーション ([src/backend](../../../src/backend)、
[src/frontend](../../../src/frontend)、[README.md](../../../README.md)) を分析してください。
現状の機能セットに不足しているギャップを特定し、アプリ拡張のための
ビジネス要件、エピック、ユーザー向けフィーチャーの優先度付きバックログを作成してください。
```

**期待される出力:**
- 既存機能のサマリー (Phase 1 MVP)
- 特定されたギャップ (例: 認証、通知、レポート、分析)
- BRD のゴール、エピック、フィーチャーにマッピングされた優先度付きバックログ
- HLD/Dev エージェントに渡せる、受け入れ志向のフィーチャーステートメント

---

## Step 6 — アプリ拡張のための新しいユーザーストーリーを作成

BRD エージェントの出力から、上位ストーリーを GitHub Issue として作成します。バックログのシード例:

| # | ストーリー | 優先度 |
|---|---|---|
| 1 | 保険契約者として、ユーザー名とパスワードでログインし、自分の契約を非公開に保ちたい | 高 |
| 2 | 保険契約者として、クレームのステータスが変わったらメール通知を受け取り、状況を把握したい | 高 |
| 3 | 管理者として、クレームのトレンドと承認率を見るレポートダッシュボードで運用を監視したい | 中 |
| 4 | 保険契約者として、自分の契約の PDF サマリーをダウンロードし、オフラインで参照したい | 中 |
| 5 | 管理者として、ステータスやプラン種別で契約を検索・フィルタし、効率的に管理したい | 中 |
| 6 | 保険契約者として、クレームのステータス変更タイムラインを見て、審査プロセスを理解したい | 低 |

Copilot Chat から GitHub MCP サーバー経由で Issue を作成:

```
@github 以下のユーザーストーリーについて、im-naga-ghas/insure-well-agentic-sdlc-ghe に
適切なラベル (enhancement、frontend、backend、auth) を付けた GitHub Issue を作成してください:
[ストーリー一覧を貼り付け]
```

## Step 7 — クラウドエージェントまたは Dev エージェントへストーリーを割り当て

バックログからストーリーを 1 つ選択します (例: *ストーリー 1 — ユーザー認証*)。

**オプション A — Dev エージェント (フィーチャー実装):**

```
@5.SDLC Dev Agent InsureWell のユーザー認証 (ストーリー #1) を実装してください。
- Spring Boot バックエンドに Spring Security ベースのログインを追加
- JPA/H2 に適したユーザーモデルと永続化戦略を導入
- React フロントエンドに認証済み UI フロー (サインイン/サインアウト) を追加
- クレームと保険契約管理アクションを認証/認可で保護
- デモ用にローカル既定の管理者ユーザーをシード
- 認証ワークフロー用のバックエンド/フロントエンドテストを追加
```

**オプション B — クラウドエージェント (インフラ/デプロイ):**

```
@cloud-agent InsureWell のデプロイ構成を作成してください。
- Spring Boot バックエンドと React フロントエンドの Dockerfile
- ローカル/デモ環境用に Web アプリと API を同時起動する docker-compose.yml
- main への push 時にコンテナイメージをビルド・公開する
     GitHub Actions ワークフロー (.github/workflows/deploy.yml)
- Spring プロファイル、API ベース URL、デプロイ構成のための
     環境変数およびシークレット管理
```

エージェントはフィーチャーブランチを作成し、変更を実装して Pull Request をオープンします。

---

## Step 8 — QA エージェントでテスト追加、CI 失敗分析、カバレッジ確認

Dev またはクラウドエージェントが PR をオープンしたら、QA エージェントを呼び出します:

**テスト追加:**
```
@6.SDLC Test Agent ストーリー #1 (ユーザー認証) の PR をレビューしてください。
以下をカバーするテストを生成してください:
- ログイン成功と認証済み API アクセス
- 誤ったパスワードでのログイン失敗
- 未認証時の保護されたルート/アクションへのアクセス
- ログアウト/セッション無効化の挙動
バックエンドテストは `src/backend/src/test/` 配下、
フロントエンドテストは適切な場所として `src/frontend/src/` 配下に追加してください。
```

**CI 失敗の分析:**
```
@6.SDLC Test Agent PR #<番号> の CI パイプラインが失敗しています。
GitHub Actions のログを分析し、根本原因を特定して
修正案または修正そのものを提示してください。
```

**コードカバレッジの確認:**
```
@6.SDLC Test Agent バックエンドとフロントエンドのテストスイートをカバレッジ付きで実行し、
カバレッジ 80% 未満のモジュールやブランチを洗い出してください。
不足を埋める追加のテストケースを提案してください。
```

---

## Step 9 — レビューエージェントで PR をレビュー

テストが通ったら、オープン中の PR に対してレビューエージェントを呼び出します:

```
@review-agent ユーザー認証フィーチャーの PR #<番号> をレビューしてください。
以下を確認:
- セキュリティ問題 (パスワードハッシュ化、セッション管理、CSRF/CORS、認可漏れ)
- 既存の Spring Boot/React コードベースとのコードスタイルの一貫性
- エラーハンドリングやエッジケースの欠落
- ドキュメントとインラインコメント
- OWASP Top 10 の懸念点
PR にインラインコメントを投稿し、サマリー判定 (Approve / Request Changes) を出してください。
```

レビューエージェントは GitHub MCP サーバー経由で構造化されたレビューコメントを Pull Request に直接投稿します。

---

## デモフロー全体のサマリー

```
クローン & 起動  →  GitHub MCP 有効化  →  カスタムエージェント追加
     ↓
HLD エージェント (アーキテクチャ)  →  BRD エージェント (要件・バックログ)
     ↓
GitHub Issue 作成 (ストーリー)  →  Coding/クラウドエージェントへ割当
     ↓
QA エージェント (テスト、CI 分析、カバレッジ)  →  レビューエージェント (PR レビュー)
     ↓
PR をマージ  →  次のストーリーで繰り返し
```
